🔧 关键技术选型

  • Godot 4.3+：开源 2D 引擎，原生支持 WebAssembly 导出，文件体积比 Unity WebGL 轻量 50% 以上
  • react-godot-bridge：React 与 Godot 的双向事件驱动通信桥，支持 postMessage 跨 iframe 通信
  • FastAPI：Python 异步 Web 框架，原生支持 WebSocket 与 Server-Sent Events

  • Celery + Redis：异步任务队列，每个 Agent 对应独立 Worker Pool

💡 价值与意义

📈 效率提升

QuantOffice 通过"空间隐喻"将抽象的量化交易流程映射为具象的办公室场景，将策略监控效率提升 40% 以上。

核心指标改善：
• 策略异常发现时间：5分钟 → 30秒
• 多策略监控窗口：5+ 个 → 1个浏览器标签页
• 新策略上线学习成本：2小时 → 10分钟

🌍 社会价值

量化交易长期以来被外界误解为"神秘的赚钱机器"，QuantOffice 的像素办公室设计将量化交易的完整工作流透明化、游戏化，大幅降低公众理解门槛。

应用场景：
• 教育培训：学生通过"观察办公室里的角色在做什么"来学习量化交易全流程
• 投资者教育：个人投资者可直观了解"我的钱是怎么被管理的"
• 行业透明化：可视化呈现有助于提升整个量化行业的信任度与普及率

🎮 创新亮点

  • 拟物化设计：参考腾讯 Marvis 的像素办公室，将抽象的多 Agent 协作具象为"看得见的工作场景"
  • 游戏化交互：点击角色即可下达指令、调整参数，像玩模拟经营游戏一样管理交易策略
  • 回测时间旅行：支持"回放"历史某一天的策略执行过程，像看电影一样复盘
  • 风险可视化：RiskAgent 的红灯警报、屏幕闪烁等效果让风险感知从"读数字"变为"看动画"

📋 创意介绍

🎯 想解决什么问题

当前量化交易平台的操作界面普遍枯燥、数据堆砌严重，交易员需要在多个终端之间切换，难以直观感知策略执行状态与风险敞口。QuantOffice 将枯燥的量化交易流程转化为可视化的像素风格"虚拟办公室"，让交易员像指挥一支专业团队一样管理量化策略，大幅降低认知负荷与操作门槛。

💡 灵感来源

灵感来源于腾讯 Marvis（马维斯） 的像素风格多 Agent 办公室设计。马维斯通过"1+5 Agent 协作架构"将系统级 AI 操作具象化为一个虚拟办公场景，每个 Agent 都有独立的工位和可视化状态。

我们认为这种"拟物化 + 游戏化"的设计理念与量化交易场景高度契合——量化交易本身就是多角色协同（数据、策略、风控、执行、报告）的复杂系统，用像素办公室的形式呈现，既能让专业交易员获得沉浸式掌控感，也能让新手快速理解量化交易的全流程。

📦 产品形态

QuantOffice 是一款基于 Godot 引擎 开发的 2D 像素风格量化交易可视化系统。前端以 React 组件形式嵌入 Godot 导出的 WebAssembly 像素办公室场景，后端采用 FastAPI + Python 架构，底层量化执行引擎集成 axon_quant。

用户通过浏览器即可访问一个活生生的"量化交易办公室"：首席交易员坐镇中央，数据分析师、策略研究员、风控官、执行交易员、报告专员各司其职，实时显示行情数据、策略状态、风险指标与交易执行进度。

👥 目标用户及痛点

🎯 核心用户群体

  • 个人量化交易者：有编程基础，希望降低策略监控成本
  • 小型量化私募团队（3-10人）：需要轻量级协同与可视化工具
  • 量化交易教育培训机构：需要直观的教学演示系统
  • 技术爱好者：对量化交易与游戏化设计感兴趣的开发者

📍 使用场景

  • 日常策略监控：开盘期间实时观察多个策略的运行状态与健康度
  • 策略回测审查：查看历史回测的绩效报告与可视化分析
  • 风险事件响应：当风险指标触发阈值时，通过像素办公室的"警报动画"第一时间感知
  • 团队协同复盘：多人同时在线查看办公室状态，讨论策略调整
  • 量化交易教学：向学生/客户演示量化交易的完整工作流

😣 当前痛点

  • 信息过载：传统量化终端以文本和表格为主，策略一多就眼花缭乱
  • 缺乏状态感知：策略运行是"黑盒"，只能看日志判断是否正常
  • 切换成本高：数据获取、策略编写、回测、风控、下单分散在不同工具中
  • 学习曲线陡峭：新人难以理解量化交易的全链路流程
  • 情绪隔离：纯数据界面缺乏"在场感"，长时间监控容易产生疲劳

🔌 QuantCell 插件化双模式架构

🎯 双模式设计目标

参考 WalletMonitor 的双模式设计，QuantOffice 支持独立运行和QuantCell 插件加载两种模式，两种模式共享 100% 业务逻辑代码。

独立模式：面向个人交易者，`python run.py` 一键启动完整服务
插件模式：面向 QuantCell 生态用户，ZIP 上传或 Git 安装后由 QuantCell 统一管理

🏗️ 双模式目录结构

QuantOffice/
├── run.py                          # 独立运行入口
├── manifest.json                   # 插件清单（插件模式必需）
├── quant_office/
│   ├── app.py                      # 独立模式：完整 FastAPI 应用工厂
│   ├── plugin.py                   # 插件模式：PluginBase 子类外壳
│   ├── core/                       # 核心业务（共享）
│   │   ├── engine_adapter.py       # axon_quant 适配器
│   │   ├── agent_scheduler.py      # Agent 调度器
│   │   ├── websocket_manager.py    # WebSocket 管理
│   │   └── event_publisher.py      # EventBus 事件发布
│   ├── agents/                     # Agent 业务逻辑（共享）
│   ├── api/                        # API 路由（共享）
│   ├── data/                       # 数据层（共享）
│   ├── services/                   # 服务层（共享）
│   └── godot/                      # Godot WASM 导出资源
├── frontend/
│   └── src/plugins/quant-office/   # 前端插件入口
│       ├── index.tsx               # 菜单/路由注册
│       ├── manifest.json           # 前端插件清单
│       └── components/             # 插件页面组件
└── godot_project/                  # Godot 源工程

⚡ 核心设计原则（参考 WalletMonitor）

  • 业务逻辑与框架完全解耦：core/、agents/、api/、data/ 等模块不依赖任何插件框架代码，两种模式 100% 复用
  • 独立模式提供完整上下文：app.py 创建完整 FastAPI 应用（CORS、中间件、数据库、根路由、健康检查），路由前缀 /api
  • 插件模式仅提供"适配外壳"：plugin.py 只负责继承 PluginBase、创建 APIRouter(prefix="/api/plugins/quant-office")、复用业务路由
  • 数据隔离：插件模式使用插件数据目录 backend/plugins/quant-office/data/，避免污染源码目录
  • 配置共享：config.py、logging_config.py 被两种模式共同使用

📦 QuantCell 插件安装方式

  • ZIP 上传：前端插件管理页拖拽上传，后端自动校验 manifest.json、移动目录、写入数据库、热加载
  • Git 克隆：提供 Git URL，后端执行 git clone --depth 1，超时 120 秒
  • 手动放置：直接将插件目录放入 backend/plugins/，系统启动时自动扫描
打包命令：
python plugin_packer.py quant_office/ → quant-office-1.0.0.zip

📡 事件驱动桥接

axon_quant 使用 crossbeam-channel bounded 100K 事件队列（零锁设计），QuantOffice WebSocket Manager 直接订阅：

from axon_quant.core import EventBus

class QuantOfficeEventBridge:
    def __init__(self, ws_manager):
        self.ws_manager = ws_manager
        self.event_bus = EventBus.new()
    
    async def subscribe(self):
        self.event_bus.subscribe("order_filled", self._on_order_filled)
        self.event_bus.subscribe("risk_alert", self._on_risk_alert)
        self.event_bus.subscribe("backtest_complete", self._on_backtest_complete)
    
    async def _on_risk_alert(self, event):
        # RiskAgent 红灯警报 → 前端像素办公室闪烁
        await self.ws_manager.broadcast({
            "type": "risk_alert",
            "agent": "RiskAgent",
            "severity": event.severity,
        })
